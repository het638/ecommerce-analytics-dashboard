1. PROJECT OVERVIEW
----------------------------------------------------------------

This is an end-to-end data analytics project that simulates a
real-world e-commerce business scenario. The goal is to analyze
sales performance, customer behavior, and product trends using
a complete data pipeline — from raw data generation to an
interactive business dashboard.

This type of project is used by companies to answer questions like:
  - How much revenue did we make this month vs last month?
  - Which products and categories are selling the most?
  - Which customers are loyal and which are at risk of leaving?
  - Which cities generate the most revenue?
  - How do customers prefer to pay?

2. PROBLEM STATEMENT
----------------------------------------------------------------

A business has thousands of orders, customers, and products.
The raw data is spread across multiple tables and is difficult
to read or understand directly. We need to:

5. DATABASE DESIGN (Star Schema)
----------------------------------------------------------------

We designed a normalized relational database with 5 tables:

  CUSTOMERS
  - customer_id (primary key)
  - customer_name, email, city, state, country
  - registration_date
  - Total: 2,000 customers

  PRODUCTS
  - product_id (primary key)
  - product_name, category, price, cost
  - 8 categories: Electronics, Clothing, Books, Sports,
    Home & Garden, Toys, Beauty, Food & Grocery
  - Total: 48 products

  ORDERS
  - order_id (primary key)
  - customer_id (foreign key → customers)
  - order_date, status (delivered/cancelled/returned)
  - Total: 15,000 orders

  ORDER_ITEMS
  - order_item_id (primary key)
  - order_id (foreign key → orders)
  - product_id (foreign key → products)
  - quantity, unit_price, discount
  - Total: 44,816 order items

  PAYMENTS
  - payment_id (primary key)
  - order_id (foreign key → orders)
  - payment_date, amount, method
  - Methods: Credit Card, Debit Card, UPI, Net Banking, COD
  - Total: 11,879 payments

  WHY THIS DESIGN?
  - Avoids data duplication
  - Each piece of information is stored only once
  - Tables are linked through foreign keys
  - This is called a "Star Schema" — industry standard for
    data warehousing and analytics

6. DATA PIPELINE (ETL Process)
----------------------------------------------------------------

ETL stands for Extract, Transform, Load — the standard process
for moving data from source to database.

  STEP 1 — EXTRACT (generate_data.py)
  - Generated 2,000 realistic customers across 8 Indian cities
  - Generated 48 products across 8 categories with real prices
  - Generated 15,000 orders over 3 years (2022-2024)
  - Generated 44,816 order items with quantities and discounts
  - Saved all data as CSV files

  STEP 2 — TRANSFORM (load_data.py)
  - Cleaned customer names (proper capitalization)
  - Standardized email addresses (lowercase)
  - Converted data types (strings to numbers where needed)
  - Removed duplicates using INSERT IGNORE

  STEP 3 — LOAD (load_data.py)
  - Connected to MySQL database
  - Cleared existing data to avoid duplicates
  - Loaded all cleaned CSV data into MySQL tables
  - Used batch insert for performance (faster than row-by-row)

7. SQL ANALYTICS VIEWS
----------------------------------------------------------------

We created 7 SQL views — these are pre-written queries saved
in the database that can be reused by any tool (Power BI, Python).

  VIEW 1: vw_monthly_revenue
  - Shows revenue, orders, and customers per month
  - Used for: revenue trend analysis
  - Key insight: track growth or decline month by month

  VIEW 2: vw_category_performance
  - Shows revenue, units sold, and profit per product category
  - Used for: identifying best and worst performing categories
  - Key insight: Electronics is the top revenue category

  VIEW 3: vw_top_products
  - Shows best selling products ranked by revenue
  - Used for: product performance analysis
  - Key insight: Smartphone X12 and Wireless Earbuds are top sellers

  VIEW 4: vw_customer_cohorts
  - Groups customers by the month they first purchased
  - Tracks how much each group spends over time
  - Used for: customer retention analysis
  - Key insight: identifies which months acquired most valuable customers

  VIEW 5: vw_rfm
  *** This is the most advanced analytics view ***
  RFM stands for Recency, Frequency, Monetary
  - RECENCY: how recently did the customer buy? (lower = better)
  - FREQUENCY: how many times did they buy? (higher = better)
  - MONETARY: how much did they spend? (higher = better)
  Each customer gets a score 1-5 on each dimension.
  Then they are grouped into segments:
    - Champions: bought recently, buy often, spend most
    - Loyal Customers: buy regularly, good spenders
    - At Risk: used to buy often but haven't recently
    - Lost: haven't bought in a long time
    - Can't Lose Them: big spenders who are becoming inactive
    - Recent Customers: bought recently but not often yet
    - Potential Loyalists: recent buyers with good frequency
  WHY RFM? This is a real technique used by marketing teams
  to decide who to target with promotions and discounts.

  VIEW 6: vw_payment_methods
  - Breakdown of revenue and transactions by payment method
  - Used for: understanding customer payment preferences

  VIEW 7: vw_geo_revenue
  - Revenue and orders breakdown by city and state
  - Used for: identifying top markets geographically
  - Key insight: Delhi generates the highest revenue

11. HOW TO RUN THE PROJECT
----------------------------------------------------------------

  REQUIREMENTS
  - Python 3.12+
  - MySQL Server 9.7+
  - Power BI Desktop

  SETUP (first time only)
  1. Open config.py and set your MySQL password
  2. Run: python setup.py
     This will create database, generate data, load data,
     and create all analytics views automatically.

  RUN STREAMLIT DASHBOARD
  - Run: python -m streamlit run dashboard.py
  - Open browser: http://localhost:8501

  OPEN POWER BI DASHBOARD
  - Open dashboard.pbix in Power BI Desktop
  - All charts will load automatically from MySQL
